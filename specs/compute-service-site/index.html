Design Spec tinycloud-node specs/compute-service.md decisions applied · 2026-07-10

Compute over data you never hand over.

The TinyCloud compute service runs functions against user-owned space data under capability-based least privilege. A WASM function is stored as content-addressed data; whoever invokes it needs only the right to run it — the function carries its own cryptographic grant to the data it touches, bound to the exact bytes of its code.

Layer 1 — Who can run

The invoker presents a tinycloud.compute/execute delegation on the function resource. The node verifies the chain cryptographically — resource containment, ability matching — with the same machinery as every other service.

Layer 2 — What the routine touches

The routine holds its own delegation (e.g. tinycloud.kv/get on <space>/kv/calendar/), bound at deploy time to the function's content CID. Data access flows back through /invoke under the routine's grant — never the invoker's authority. Change the code, and the grant is void.

01Overview

Least-privilege function execution over space data

TinyCloud spaces already expose KV, SQL, and DuckDB services behind a single capability-gated endpoint. The compute service adds a fourth: deploy a WASM function into a space, then let holders of an execute capability run it — without ever granting them access to the data the function reads or writes.

Design goals:

  • Programs are data. Functions are stored content-addressed; the function's identity is its content CID. Named and versioned like any other space artifact.
  • No new endpoint. Everything rides the node's single POST /invoke; dispatch is by the capability's service string, exactly like KV / SQL / DuckDB.
  • Two-layer permissioning. Invocation rights and data rights are separate capability chains. The invoker never needs the underlying data permission.
  • Pluggable execution backends behind one interface — in-node Wasmtime first, Cloudflare Workers second, TEE-quote and ZK verifiers as documented future plugs.
  • Authorization stays home. Data authorization always happens at the data-owning node, regardless of where compute runs. Backends never get raw data access.

Out of scope: multi-node orchestration (running the same routine across several people's nodes and aggregating into a shared space) is handled by layers above, e.g. Smithers. See §13.

02Core Model

Programs are data

Three abilities, all pre-reserved in the protocol's capability vocabulary (whitepaper Appendix J) and landing in the node's capabilities.json SSOT as status: "reserved" before activation:

AbilityResourceWhat it does
tinycloud.compute/deploy privileged <space>/compute/<name> Stores a WASM binary content-addressed and carries the routine's data grant D_fn. Function identity = content CID; named + versioned via artifact revisions. Re-deploying identical bytes is idempotent by CID.
tinycloud.compute/execute <space>/compute/<name> Invokes a function: function ref + inputs (inline JSON or KV refs), with an optional exact content_cid pin against re-deploy races. Output returned inline or written to a KV path.
tinycloud.compute/list <space>/compute/ Enumerates deployed functions (name, CID, revision).
Decided — deploy tier: privileged, no compute/admin

compute/deploy is the privileged, code-mutating ability — the analog of tinycloud.sql/schema. Standard-tier session grants exclude it: a browser session may hold execute and list, but deploying executable code requires an explicitly elevated grant. No compute/admin URN is minted now; a future one would implies deploy (mirroring sql/admin ⊃ sql/schema) only once a real admin-only surface exists. Corollary (MUST): the SDK standard session grant must enumerate compute/execute + compute/list and never compute/* — once the concretes go active the wildcard implies deploy, so copying the existing sql/* grant habit would silently hand every browser session deploy rights.

The resource URI needs zero parser changes: ResourceId is already <space>/<service>/<path>, and compute is just another service string. Containment (extends()) does the same space / service / path-prefix check it does for KV — a grant on <space>/compute/ (trailing slash) is space-wide compute authority; <space>/compute/report-generator is single-function authority.

Function storage reuses the node's existing artifact repository (DatabaseArtifactRepository — the same store SQL and DuckDB persist through), keyed (service = "compute", space, name) with a computed content CID and a monotonically increasing revision. If output is written to a KV path, normal KV read delegations then govern who can read it — compute introduces no parallel read path.

03The Novel Part

Two-layer permissioning

Every other capability system answers one question: may this principal perform this action on this resource? Compute needs two answers at once — and keeps them in separate chains.

Layer 1 — who can run invoker's chain

The invoker holds a tinycloud.compute/execute delegation on <space>/compute/<fn>. The node verifies the delegation chain cryptographically — resource containment plus ability matching against the persisted chain — using the existing invocation validation path, unchanged. Running a function is just another capability-gated action.

Layer 2 — what the routine touches routine's own grant

The routine holds its own delegation, D_fn — for example tinycloud.kv/get on <space>/kv/inputs/ and kv/put on outputs/ — minted by the deployer at deploy time, shipped inside the deploy request, and bound to the function's content CID. When the running function needs data, the host mediator constructs an internal invocation signed by the routine's key, citing D_fn as parent, and runs it through the normal invocation validate/save path — entering via core process(), not back through the HTTP route.

The routine's execution identity is deterministic — derived by the node from the function's canonical resource URI under a versioned domain prefix, via the existing dstack hierarchical key derivation: 

resource_uri = ‹space›/compute/‹function_cid›   // the ResourceId display form (§2)
routine_did  = did:key( get_key("tinycloud/compute-key/v1/" + resource_uri) )

Using the canonical ResourceId display string means one canonical spelling is shared by the authorizer, the binding caveat, and the derivation — no second serialization to keep in sync — and the versioned compute-key/v1/ prefix is domain-separated, so a future v2 scheme cannot collide with existing identities. The private key is derived inside the TEE and never leaves it, so only this node, running this exact function CID in this exact space, can act as the routine. The space component is mandatory, not cosmetic: identical WASM deployed in two spaces shares one CID, and a CID-only identity would open a cross-space confused deputy — per-(space, function) derivation makes cross-space citation cryptographically impossible rather than merely policy-blocked. The public routine_did is not computable client-side (if it were, the private key would be too) — the deployer obtains it via a read-only RoutineDid handshake before deploy (§4). (Non-TEE classic mode runs the same derivation off the node's static key material; the trust statement weakens to "the node".)

Blocking before activation — space-name validation

The derivation string's uniqueness requires that distinct (space, function) pairs never collide into one string. CIDs are base32 (delimiter-free) — safe. But space Name validation is currently a stubbed // TODO in tinycloud-auth: before compute activates, space names MUST be validated to exclude the URI delimiters (/, :, #, ?) — or the space component MUST be hashed (fixed-width, delimiter-free) before it enters the derivation string. Tracked as a test obligation (§13).

Consequences worth stating plainly:

  • The invoker never needs the underlying data permission. Alice can let Bob run her calendar digest without Bob ever holding kv/get on her calendar.
  • Changing the code voids the grant. A new WASM binary is a new CID; the old CID-bound D_fn no longer applies. Code changes cannot silently inherit data access.
  • No new authorization-engine code — Layer 2 rides the same delegatee-match and chain-containment checks as every user invocation. But the host mediator carries one normative MUST: it echoes D_fn's caveat map verbatim on every internal invocation (byte-equality containment rule) — a caveat-less internal invocation fails closed on the first host call. The echo propagates down the Cloudflare chain (D_worker and Worker callbacks) too.
Layer 1 — invoker's execute chain Layer 2 — routine's CID-bound data grant
Decided — D1: routine identity = TEE-derived key from (space, function CID)

Chosen for no secret at rest and for binding data access to the function identity, the space, and the node identity (attestable via /attestation). Identity enforcement is cryptographic — only the execution of function_cid in that space can derive the key matching D_fn.delegatee — reusing did_principal_matches and the existing chain walk, with no new runtime "am I inside function X?" check. Cost accepted: a redeploy on a different node yields a different routine DID, so D_fn is re-minted per node.

Deployment risk — dstack key stability (verify empirically)

D1 assumes dstack::get_key(path) returns a stable key for a given path across CVM redeploys. That stability is a known open question in this stack — a prior DID-drift incident in OpenCredentials showed dstack-derived key material shifting across deploys, which would change routine_did and silently invalidate every D_fn bound to the old identity.

Before production reliance: derive routine_did for a fixed (space, function_cid), redeploy the CVM, re-derive, assert equality — the read-only RoutineDid handshake (§4) is exactly this probe.

Compare-on-execute tripwire: before running a function the node re-derives the key and compares against D_fn.delegatee; a mismatch fails with the distinct routine-identity-rotated error (409/503, not a generic 403), so a seed rotation surfaces as itself, not as a buried authorization bug.

Recorded fallback if derivation proves unstable: a freshly generated per-deploy routine key persisted encrypted alongside the artifact — a localized ComputeService change with no impact on the wire format, the binding caveat, or the authorization layers.

Decided — D2: the binding is a self-describing caveat on D_fn

The function_cid → D_fn link is not an internal node-side table. It is a caveat on the deploy-time delegation itself:

{ "computeFunctionBinding": { "functionCid": "<function_cid>" } }

An internal table would be out-of-band state, unverifiable from the event graph. The caveat keeps the binding auditable (anyone replaying the delegation events sees exactly which function each D_fn authorizes) and portable (a node migration carries the binding inside the signed delegation). It composes with D1 rather than overlapping it: the derived key makes the binding unforgeable; the caveat makes it auditable. The execute path selects D_fn by (space, functionCid) — the space of the outer invocation, never the CID alone — verifies every capability resource lives in that space, and cites all matching valid D_fns as parents (graceful degradation when an older grant is revoked mid-life). On re-deploy the superseded D_fn is revoked.

Two caveat namespaces — do not conflate them

Compute introduces caveats in two different places, on two different delegations, doing two different jobs:

CaveatLives onJob
computeFunctionBinding layer 2 The routine's data grant D_fn (a delegation) Self-describing record binding the grant to one functionCid. Auditable from the event graph; enforced by the derived-key identity.
ComputeCaveats layer 1 The invoker-facing compute/execute delegation chain Attenuates who can run what, how hard: functions allowlist, maxDuration, maxMemory, inputs schema (§8).
04Sequence

Deploy flow

Deployment starts with a read-only handshake: the client hashes the WASM locally (the CID convention is public), then asks the node for the routine identity — routine_did is derived from the node's TEE-internal seed and is not computable client-side. The node returns the public DID, no secret exposed, no side effects. The client mints D_fn against it (with the computeFunctionBinding caveat naming the CID) and ships binary + grant in one deploy request; the node processes D_fn through the standard /delegate path atomically with the artifact save and quota accounting. Deploy authorization is the privileged tier (§2) — standard-tier sessions cannot do this.

05Sequence

Execute flow — in-node WASM WasmtimeBackend

The default backend runs the function inside the node under Wasmtime. Caveats map directly onto runtime controls — maxDuration to epoch interruption, maxMemory to StoreLimits, plus fuel metering as a belt-and-braces CPU budget — and every data access is a host function call mediated by the node, re-entering /invoke under the routine's grant. The guest has no WASI network or filesystem; its imports are exactly the mediator surface.

That closed surface enables the execution manifest (normative on this backend): the mediator journals every host call — (resource, ability, bytes in/out, output destination) — plus the capabilities granted by D_fn versus those actually exercised. With no unmediated egress, the manifest is ground truth: nothing leaves except through a logged host call or the returned output. Granted-but-unexercised capabilities are the deployer's concrete scope-down signal for the next deploy. The manifest rides in the outcome metadata (and MAY additionally persist to a KV audit path under the routine's own grant, opt-in via node config).

06Sequence

Execute flow — Cloudflare backend trust-the-deployment

The same entry point, a different run(). The Worker generates a fresh ephemeral keypair per execution (never reused) and returns its public key; the node mints D_worker for that pubkey — a child of D_fn, attenuated in resource/ability only, echoing the binding caveat, with a TTL of roughly maxDuration + slack under a config ceiling (and capped by D_fn's own expiry). The routine's private key never leaves the node, and the worker key never transits Cloudflare. Authorization still happens at the data-owning node: callbacks are validated against the attenuated chain, so the Worker cannot exceed the routine's data caps. On completion, timeout, or error the node revokes D_worker — the real kill switch (revocations are checked per invocation); the TTL is only the crash backstop, and revoking D_fn itself is the deployer's panic button, killing every live D_worker transitively.

Egress is mediated too: routines deploy into a Workers for Platforms dispatch namespace with an Outbound Worker interposed on all fetch() from the routine — a default-deny allowlist whose sole default entry is the node's callback endpoint, with the full egress log returned in the run evidence (folded into the execution manifest, which on this backend is a lower bound raised toward ground truth for fetch). Enabling it disables raw TCP connect(); the known DO/mTLS fetch bypass is closed by construction because the node authors the deploy and grants routines no DO/mTLS bindings.

Trust model, stated honestly

The headline caveat is data egress. The wasmtime backend has no exfiltration channel — no net, no fs, data flows only to the output or to D_fn-granted KV paths. A bare Cloudflare Worker has unrestricted outbound fetch, so without mitigation everything readable under the grant must be treated as disclosed to the function author and to Cloudflare. With the Outbound Worker in place (default-deny allowlist, node callback as the sole default entry, raw TCP connect() disabled, DO/mTLS bypass closed by construction), function-author exfiltration is CF-enforced away and the rule softens — not disappears — to: "disclosed to Cloudflare; the function author is confined by the CF-enforced egress policy." Cloudflare runs the interceptor and sees plaintext (the input also transits and rests on its infra). Scope read grants on that basis; on the in-node wasmtime backend there is no such disclosure. This confidentiality posture, not the resource-limit gap, is the decisive fact when choosing backends.

Code identity is a bookkeeping claim, not content-addressed. On wasmtime the node hashes the bytes it is about to run; on CF the CID binds the grant to what the node uploaded, not what runs — anyone with Cloudflare account access (including Cloudflare) can swap the script behind the route while D_worker is valid. The Cloudflare account credentials join the TCB. verification: { mode: "trusted-deployment", backend: "cloudflare" } carries exactly this meaning.

Resource caveats are hints. maxMemory, maxDuration, and CPU cannot be enforced by the node on CF. What survives the boundary: Layer-1 and Layer-2 authorization, the CID-bound grant via D_worker, and capability-checked data responses. Callers who need enforced bounds or confidentiality use the in-node WasmtimeBackend.

07Architecture

Pluggable execution backends

All backends implement one interface (spec §9):

#[async_trait]
pub trait ExecutionBackend: Send + Sync {
    /// Execute `wasm` against `input`, mediating host imports through `host`
    /// (which issues the routine's data invocations under D_fn), enforcing `caveats`.
    async fn run(&self, function_cid: &str, wasm: &[u8], input: ComputeInput,
                 caveats: &ComputeCaveats, host: &dyn HostMediator)
        -> Result<ComputeOutput, ComputeError>;

    /// Post-process / package the output (e.g. write to a KV path).
    async fn output(&self, out: ComputeOutput, sink: OutputSink, host: &dyn HostMediator)
        -> Result<serde_json::Value, ComputeError>;

    /// Produce (or check) an execution proof. Trivial for the trusted in-node
    /// backend; the plug point for TEE-quote and ZK verification.
    async fn verify(&self, ctx: &ExecutionContext) -> Result<Verification, ComputeError>;
}

Invariant

Regardless of backend: authorization always happens at the data-owning node. Backends receive a grant token, never raw data access. The weakest backend can waste compute; it cannot widen data reach.

08Constraints

ComputeCaveats & enforcement

Delegations carrying compute abilities may attenuate with ComputeCaveats (camelCase on the wire, per whitepaper Appendix J). This is the invoker-facing caveat set — distinct from the computeFunctionBinding caveat that lives on D_fn (§3). 

#[derive(Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct ComputeCaveats {
    pub functions:    Option<Vec<String>>,       // allowed function names (allowlist)
    pub max_duration: Option<u64>,               // maximum execution time (ms)
    pub max_memory:   Option<u64>,               // maximum memory usage (bytes)
    pub inputs:       Option<serde_json::Value>, // input validation schema
}
Fail-closed: caveat source is the chain, not the facts

Following the SQL service's W1 lesson, the enforced values — especially the functions allowlist — are read from the validated delegation chain. Caveats supplied in invocation facts (computeCaveats) are a non-authoritative fallback only. An invoker cannot widen or drop the allowlist by editing the invocation envelope. Numeric caveats are validated against sane ceilings on ingest and applied without silent truncation. The byte-equality rule bites Layer 1 too: an invoker under a ComputeCaveats-carrying grant must echo the chain's caveat map verbatim on the invocation (the SDK compute helper copies it, same as the SQL constrained-statements flow) or the invocation is rejected before the handler runs.

09State

Function lifecycle

A function's life is anchored to its content CID. Deploying new bytes under the same name produces a new CID and a fresh lifecycle; the old CID's grant does not carry over.

A function that is Deployed but never grant-bound is still executable if it needs no data access at all — pure functions over inline inputs skip Layer 2 entirely.

10State

Invocation lifecycle

Each execution request moves through verification, dispatch, and delivery. Replay protection is nonce-based, as everywhere else in the protocol: the outer compute/execute envelope passes through the invocation replay cache, and a previously seen envelope is rejected before any chain work happens. The routine's internal invocations are freshly minted per execution (fresh nonces), so running the same function twice is two distinct envelopes — allowed — not an accidental replay. Execute idempotency is a function property, not a service guarantee; the service does not implicitly memoize results.

11Protocol Surface

Wire format & capability registry

Compute follows the SQL/DuckDB precedent byte for byte: /invoke filters presented capabilities for the compute service, deserializes the body into a tagged ComputeRequest enum (action, snake_case), and answers through new InvocationOutcome arms. Compute is disabled by default behind a compute cargo feature: without it, a compute-service capability returns 501 Not Implemented ("Compute support is not enabled on this node") — the same gating the DuckDB feature flag uses — and with it, /version advertises "compute" in its features list for client feature-detection. 

// request body over POST /invoke — final shapes per spec §7.2
{ "action": "execute",

  "content_cid": "bafk…",              // optional pin against a re-deploy race
  "input":       { … },                // inline JSON input, and/or…
  "input_refs":  ["kv/inbox/x"],       // KV paths the routine reads under D_fn

12Verification

Trust & verification ladder

verify(evidence) is where backends differ most. The design names the rungs explicitly rather than pretending they are equivalent:

RungEvidenceWhat it proves
1 · Same trust domain None needed — WasmtimeBackend runs inside the node. Everything the node itself is trusted for. Hosted nodes run in dstack TEEs with a live /attestation?nonce endpoint (TDX quote, nonce-bound), so "the node" is itself attestable. Non-TEE classic mode weakens rung 1 to trusting the node's static key material.
2 · Trust the deployment Deployment identity of the Cloudflare Worker; result carries verification: { mode: "trusted-deployment" }. That some deployed code produced the output. No runtime-caveat guarantees; data reach still bounded by the callback delegation attenuated from D_fn.
3 · Remote TEE quote future TDX quote whose report_data commits to (function_cid, input_hash, output_hash), checked against the expected compose_hash. The exact runner image executed this exact function on these inputs, in genuine TEE hardware — "attest the node AND this specific execution".
4 · ZK proof future verify(proof, function_cid, inputs, outputs) — RISC Zero / SP1. This exact function, on these inputs, produced these outputs. No trust in the runner at all.

Rungs 3 and 4 are documented plugs — the Evidence type and verify() hook exist for them, but no implementation ships in v1.

Future directions noted, not built

  • Container/VM backend via the iron-proxy pattern. A future self-hosted container backend would front all egress with an iron-proxy-style mediator: default-deny outbound allowlist, per-request audit log, and credential brokering (the sandbox holds only a short-lived proxy token; the real secret is injected at the proxy boundary, never inside the sandbox). That recovers wasmtime's execution-manifest ground-truth property — every byte out passes a logged, policy-gated boundary — without an in-process WASM sandbox. No container backend is specced.
  • ZK verification is two-phase when it lands. The prover cannot run in a Worker (~128 MB isolate; current Jolt wants GPU-class hardware; a <2 GB streaming prover may fit CF Containers later). Realistic shape: execute fast, return the output marked unverified, prove asynchronously, and let verify() flip it to verified — the run → output+evidence → verify interface already supports the split. Wrinkle: Jolt proves RISC-V, not WASM, so a zk-verified function pins a RISC-V build digest in the binding caveat alongside the WASM CID. ZK fixes integrity (F7) fully in principle; it does not address confidentiality (F6) — a proof that the right code ran says nothing about where the data went. Cheap interim integrity for CF: deterministic wasmtime + random k% spot re-execution on the node.
13Boundaries

Resolved decisions, open items & scope

Resolved decided 2026-07-10

  • D1 — routine identity: deterministic TEE-derived key from the function CID (§3). No secret at rest; binds data access to function + node. Carries the dstack key-stability deployment risk — verify empirically on the target CVM before production reliance; fallback is a persisted, encrypted per-deploy key (localized change, no wire/auth impact).
  • D2 — CID binding: a self-describing computeFunctionBinding caveat on D_fn, not an internal table (§3). The derived key makes the binding unforgeable; the caveat makes it auditable and portable.
  • Deploy tier: compute/deploy is privileged (the sql/schema analog), excluded from standard-tier grants; no compute/admin URN until a real admin-only surface exists (§2).

Still open narrowed by the fable review

  • D_worker TTL tuning — the mechanics are now concrete (worker-side ephemeral keypair per execution, TTL ≈ maxDuration + slack under a config ceiling, per-execution revocation kill switch, D_fn revocation as panic button); the remaining open bit is the exact slack multiplier / ceiling default against real Cloudflare dispatch jitter.
  • Result cache — deferred, gate criterion recorded — cacheable iff D_fn grants no read capability and the request has no input_refs (a structural, checkable gate — "the function declares itself pure" is the wrong gate, declarations lie). No consumer today.
  • Binary inline outputs — deferred — the KV output path already handles binary; revisit when a consumer inlines >1 MB binaries.

Test obligations from the review

  • Caveat-echo: a non-echoing internal invocation (and CF callback) is rejected; the echoed one succeeds. Invoker-side echo likewise.
  • Cross-space isolation: identical WASM in spaces A and B yields distinct routine DIDs; a B execution cannot cite A's D_fn.
  • Space-name validation (precondition): a space name containing a URI delimiter (/ : # ?) is rejected — or the space component is hashed — so no two distinct spaces can produce the same derivation string.
  • Execution manifest: a run's returned manifest lists every host call and the granted-vs-exercised sets; an untouched granted capability shows as granted-but-unexercised (the scope-down signal).
  • Deploy atomicity: a failed D_fn leaves no artifact row and no quota entry, and vice versa.
  • Quota: deploy increments store_size; an over-quota deploy fails.
  • Rotation: re-derived key mismatch fails with routine-identity-rotated, never a generic 403.

Explicitly out of scope

  • Multi-node orchestration — running the same routine across several people's nodes and aggregating results into a shared space belongs to layers above the protocol (e.g. Smithers). The compute service is the per-node execution primitive: single-node, single-space.
  • A general FaaS control plane — autoscaling, cold-start pools, billing metering. Deferred.
