Compute over data you never hand over.

The TinyCloud compute service runs functions against user-owned space data under capability-based least privilege. A WASM function is stored as content-addressed data; whoever invokes it needs only the right to run it — the function carries its own cryptographic grant to the data it touches, bound to the exact bytes of its code.

Layer 1 — Who can run

The invoker presents a tinycloud.compute/execute delegation on the function resource. The node verifies the chain cryptographically — resource containment, ability matching — with the same machinery as every other service.

Layer 2 — What the routine touches

The routine holds its own delegation (e.g. tinycloud.kv/get on <space>/kv/calendar/), bound at deploy time to the function's content CID. Data access flows back through /invoke under the routine's grant — never the invoker's authority. Change the code, and the grant is void.

01Overview

Least-privilege function execution over space data

TinyCloud spaces already expose KV, SQL, and DuckDB services behind a single capability-gated endpoint. The compute service adds a fourth: deploy a WASM function into a space, then let holders of an execute capability run it — without ever granting them access to the data the function reads or writes.

Design goals:

  • Programs are data. Functions are stored content-addressed; the function's identity is its content CID. Named and versioned like any other space artifact.
  • No new endpoint. Everything rides the node's single POST /invoke; dispatch is by the capability's service string, exactly like KV / SQL / DuckDB.
  • Two-layer permissioning. Invocation rights and data rights are separate capability chains. The invoker never needs the underlying data permission.
  • Pluggable execution backends behind one interface — in-node Wasmtime first, Cloudflare Workers second, TEE-quote and ZK verifiers as documented future plugs.
  • Authorization stays home. Data authorization always happens at the data-owning node, regardless of where compute runs. Backends never get raw data access.

Out of scope: multi-node orchestration (running the same routine across several people's nodes and aggregating into a shared space) is handled by layers above, e.g. Smithers. See §13.

02Core Model

Programs are data

Three abilities, all pre-reserved in the protocol's capability vocabulary (whitepaper Appendix J) and landing in the node's capabilities.json SSOT as status: "reserved" before activation:

AbilityResourceWhat it does
tinycloud.compute/deploy <space>/compute/<name> Stores a WASM binary content-addressed. Function identity = content CID; named + versioned via artifact revisions.
tinycloud.compute/execute <space>/compute/<name> Invokes a function: function ref + inputs (inline bytes or KV refs). Output returned inline or written to a KV path.
tinycloud.compute/list <space>/compute/ Enumerates deployed functions (name, CID, revision).

The resource URI needs zero parser changes: ResourceId is already <space>/<service>/<path>, and compute is just another service string. Containment (extends()) does the same space / service / path-prefix check it does for KV.

Function storage reuses the node's existing artifact repository (DatabaseArtifactRepository — the same store SQL and DuckDB persist through), keyed (service = "compute", space, name) with a computed content CID and a monotonically increasing revision. If output is written to a KV path, normal KV read delegations then govern who can read it — compute introduces no parallel read path.

03The Novel Part

Two-layer permissioning

Every other capability system answers one question: may this principal perform this action on this resource? Compute needs two answers at once — and keeps them in separate chains.

Layer 1 — who can run invoker's chain

The invoker holds a tinycloud.compute/execute delegation on <space>/compute/<fn>. The node verifies the delegation chain cryptographically — resource containment plus ability matching against the persisted chain — using the existing invocation validation path, unchanged. Running a function is just another capability-gated action.

Layer 2 — what the routine touches routine's own grant

The routine holds its own delegation, D_fn — for example tinycloud.kv/get on <space>/kv/calendar/ — minted by the deployer at deploy time and bound to the function's content CID. When the running function needs data, the access goes back through /invoke as an internal invocation under the routine's grant, validated by the standard capability engine.

Consequences worth stating plainly:

  • The invoker never needs the underlying data permission. Alice can let Bob run her calendar digest without Bob ever holding kv/get on her calendar.
  • Changing the code voids the grant. A new WASM binary is a new CID; the old CID-bound D_fn no longer applies. Code changes cannot silently inherit data access.
  • No new auth machinery. Layer 2 rides the same delegatee-match and chain-containment checks as every user invocation.
Layer 1 — invoker's execute chain Layer 2 — routine's CID-bound data grant
Decision needed — D1: routine identity

Option A (proposed): routine identity is a deterministic TEE-derived key — routine_did = did:key(get_key("tinycloud/compute/" + function_cid)) via the node's existing dstack derivation. No secret at rest; the key never leaves the TEE; the deployer can compute the CID (and hence the routine DID) before deploy. Cost: a redeploy on a different node yields a different routine DID, so D_fn is re-minted per node.

Option B: a per-deploy session key generated fresh and persisted encrypted alongside the artifact. Survives node migration but adds a secret at rest.

Decision needed — D2: how D_fn binds to the CID

Table-side (leaned): the artifact store keeps function_cid → D_fn; no wire or caveat change. Caveat-side: embed function_cid as a caveat, making the binding self-describing inside the delegation itself.

04Sequence

Deploy flow

Deployment is two ordinary protocol operations: an /invoke that stores the binary, and a /delegate that mints the routine's data grant. Because the content CID is a pure function of the WASM bytes, the deployer can compute it — and therefore the routine's identity — before deploying. No chicken-and-egg.

05Sequence

Execute flow — in-node WASM WasmtimeBackend

The default backend runs the function inside the node under Wasmtime. Caveats map directly onto runtime controls — maxDuration to fuel / epoch interruption, maxMemory to store limits — and every data access is a host function call mediated by the node, re-entering /invoke under the routine's grant.

Layer 1 — invoker authorization Layer 2 — routine-grant data access
06Sequence

Execute flow — Cloudflare backend trust-the-deployment

The same entry point, a different run(). The node ships the function ref, inputs, and the routine's grant to a deployed Worker. Crucially, authorization still happens at the data-owning node: the Worker's data accesses come back as /invoke calls under D_fn, validated by the same engine. What the node cannot do is enforce runtime caveats inside Cloudflare's sandbox.

Trust model, stated honestly

With the Cloudflare backend, verify() reduces to trusting that the deployed Worker runs the code it claims to. Resource caveats (maxMemory, maxDuration) cannot be enforced by the node there. What survives the boundary: Layer-1 and Layer-2 authorization, CID-bound grants, and the guarantee that raw data never leaves the node except through capability-checked responses.

07Architecture

Pluggable execution backends

All backends implement one interface:

// provisional shape — final trait in specs/compute-service.md
trait ExecutionBackend {
    /// Start execution of a stored function under the routine's grant.
    fn run(function_cid: Cid, inputs: Inputs, grant: Delegation) -> Handle;

    /// Retrieve output plus whatever evidence the backend can produce.
    fn output(handle: Handle) -> (Bytes, Evidence);

    /// Check the evidence. Strength varies by backend — see below.
    fn verify(evidence: Evidence) -> bool;
}

WasmtimeBackend In the node process. Hosted nodes already run inside dstack TEEs. Full — fuel/epoch for maxDuration, StoreLimits for maxMemory, host-function mediation for all data access. Trivial (same trust domain). TEE deployments expose a live /attestation?nonce endpoint returning a TDX quote. Initial
CloudflareWorkerBackend A deployed Worker; node sends fn ref + inputs + routine grant. Partial — maxMemory / maxDuration not node-enforceable. Data access still capability-checked at the node. Trust-the-deployment. Initial
Remote TEE Any TEE-hosted runner. Full, if quote checks out. TDX/TEE quote checked against expected compose_hash. Documented, not built
ZK prover Anywhere — trust moves into the proof. N/A — correctness is proven, not policed. verify(proof, function_cid, inputs, outputs) — RISC Zero / SP1. Documented, not built
Invariant

Regardless of backend: authorization always happens at the data-owning node. Backends receive a grant token, never raw data access. The weakest backend can waste compute; it cannot widen data reach.

09State

Function lifecycle

A function's life is anchored to its content CID. Deploying new bytes under the same name produces a new CID and a fresh lifecycle; the old CID's grant does not carry over.

A function that is Deployed but never grant-bound is still executable if it needs no data access at all — pure functions over inline inputs skip Layer 2 entirely.

10State

Invocation lifecycle

Each execution request moves through verification, dispatch, and delivery. Replay protection is nonce-based, as everywhere else in the protocol: a previously seen invocation is rejected before any chain work happens.

11Protocol Surface

Wire format & capability registry provisional

Compute follows the SQL/DuckDB precedent byte for byte: /invoke filters presented capabilities for service == "compute", deserializes the body into a request enum, and answers through a new InvocationOutcome arm. When the node is built without the compute feature, a compute-service capability returns 501 Not Implemented — the same gating the DuckDB feature flag uses.

capabilities.json — reserved first

New service entries land as status: "reserved" in the node's capabilities.json single source of truth before activation — the same path tinycloud.vfs/* took. The registry's generator enforces that a wildcard implies every active concrete action of its service, so the reserved wildcard carries no implies until the concrete actions go active.

12Verification

Trust & verification ladder

verify(evidence) is where backends differ most. The design names the rungs explicitly rather than pretending they are equivalent:

RungEvidenceWhat it proves
1 · Same trust domain None needed — WasmtimeBackend runs inside the node. Everything the node itself is trusted for. Hosted nodes run in dstack TEEs with a live /attestation?nonce endpoint (TDX quote, nonce-bound), so "the node" is itself attestable.
2 · Trust the deployment Deployment identity of the Cloudflare Worker. That some deployed code produced the output. No runtime-caveat guarantees; data reach still bounded by D_fn.
3 · Remote TEE quote future TDX quote checked against expected compose_hash. The exact runner image executed, in genuine TEE hardware.
4 · ZK proof future verify(proof, function_cid, inputs, outputs) — RISC Zero / SP1. This exact function, on these inputs, produced these outputs. No trust in the runner at all.

Rungs 3 and 4 are documented plugs — the Evidence type and verify() hook exist for them, but no implementation ships in v1.

13Boundaries

Scope & open questions

Explicitly out of scope

  • Multi-node orchestration — running the same routine across several people's nodes and aggregating results into a shared space belongs to layers above the protocol (e.g. Smithers). The compute service is single-node, single-space.
  • A function marketplace / discovery layer — compute/list enumerates one space; anything richer is an application concern.

Open questions

  • D1 — routine identity: TEE-derived deterministic key vs persisted per-deploy session key (§3).
  • D2 — CID binding of D_fn: artifact-table side vs self-describing caveat (§3).
  • Replay-cache scope and retention for compute invocations (idempotent re-execution vs strict once-only).
  • Whether compute/deploy should be delegable below the space owner in v1.
